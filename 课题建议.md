## 论文题目

> **新质生产力冲击下的城市交通复苏与结构演化：基于纽约多源时空数据的交叉验证**

## 数据资产概览

| 数据 | 时间跨度 | 粒度 | 规模 |
|---|---|---|---|
| MTA 每日各运输方式使用情况 | 2020–2026 | 日 × 运输方式 | 中等 |
| MTA 地铁每小时刷卡记录 | 2022–2024 | 小时 × 站点 | ~8GB |
| LinkedIn 就业数据 | 2024 年为主 | 个体级（岗位/技能/行业/地区） | 130 万条 |

> **数据关系**：三份数据时间重叠窗口为 2022–2024。其中 LinkedIn 主要集中在 2024 年，与 MTA 数据做因果推断时需注意时间跨度不对称——更适合作为**截面分析**或**2024 年快照对比**，而非完整时间序列的因果检验。

## 核心概念解析

### 新质生产力 → 交通冲击

"新质生产力"行业（AI、大数据、新能源、半导体等）具有三个与传统行业不同的通勤特征：
1. **高技能集聚**：岗位集中于特定城区，通勤流向高度定向
2. **混合办公渗透**：远程/弹性工作比例高于传统行业，冲击早晚高峰结构
3. **薪资溢出效应**：高薪岗位推高周边居住成本，通勤距离可能拉长

这三条构成了新质生产力对城市交通的**冲击传导链**：岗位结构变化 → 通勤行为变化 → 交通系统负荷再分配。

### 交叉验证

三份数据各自回答同一个问题的不同侧面。如果 LinkedIn 说"A 区新质岗位密集"，那么 MTA 刷卡数据应当观测到 A 区的客流恢复更快/通勤模式更集中——用**行为数据验证文本数据**的推断，这就是交叉验证的核心逻辑。

### 第三章：数据与特征工程（核心工作量）

#### 3.1 LinkedIn 数据的 NLP 管道

**目标**：从 130 万条岗位中自动提取"新质生产力"行业标签，而非人工定义关键词。

**操作路径**：

```python
# 步骤 1：文本预处理
# - 合并 title + skills + industry 为一个字段
# - 小写化、去特殊字符、去停用词

# 步骤 2：TF-IDF 向量化（scikit-learn）
from sklearn.feature_extraction.text import TfidfVectorizer

tfidf = TfidfVectorizer(
    max_features=5000,    # 取 top 5000 词作为特征空间
    ngram_range=(1, 2),   # 既保留 "data" 也保留 "data science"
    stop_words='english'
)
job_matrix = tfidf.fit_transform(linkedin['text_field'])  # 130万 × 5000 稀疏矩阵

# 步骤 3：无监督聚类（KMeans 或 NMF 主题模型）
from sklearn.decomposition import NMF
from sklearn.cluster import KMeans

# NMF 提取主题：每个主题对应一个行业簇
nmf = NMF(n_components=20, random_state=42)
topic_weights = nmf.fit_transform(job_matrix)  # 130万 × 20

# 每个岗位分配归属主题
linkedin['topic'] = topic_weights.argmax(axis=1)

# 步骤 4：主题解释
# 取每个 topic 的 top 关键词，人工判断是否为"新质"类
# topic_3: {"ai", "machine learning", "deep learning", "pytorch", "tensorflow"} → 新质
# topic_7: {"retail", "customer service", "cashier", "inventory"} → 传统

# 步骤 5：按地理区域聚合
geo_features = linkedin.groupby('region').agg(
    new_quality_ratio=('is_new_quality', 'mean'),   # 新质岗位占比
    avg_salary=('salary', 'mean'),
    remote_ratio=('is_remote', 'mean'),
    job_diversity=('topic', 'nunique'),             # 行业多样性
    total_jobs=('id', 'count')
)
```

#### 3.2 MTA 日数据的恢复指数与结构演化

**目标**：从 MTA 日数据中提取两个核心指标——**恢复深度**（跌了多少）和**结构分化**（各运输方式的恢复差异）。

**DuckDB 聚合（入口）**：

```sql
SELECT
    date,
    transit_mode,
    SUM(ridership) AS daily_total
FROM read_csv('MTA_daily_2020_2026.csv')
GROUP BY date, transit_mode
ORDER BY date;
```

**Pandas 特征工程（后续）**：

```python
# 恢复率 = 当月日均 / 2019 年同月日均（基线）
df['year'] = df['date'].dt.year
df['month'] = df['date'].dt.month

baseline = df[df['year'] == 2019].groupby('month')['daily_total'].mean()
df['recovery_rate'] = df['daily_total'] / df['month'].map(baseline)

# 结构分化指标：各运输方式恢复率的离散度
dispersion = (
    df.groupby(['year', 'month'])['recovery_rate']
    .agg(['std', 'max', 'min'])
    # std 越大 → 结构分化越严重
)
```

**时间序列分析（statsmodels）**：

```python
from statsmodels.tsa.stattools import adfuller
from statsmodels.tsa.stattools import grangercausalitytests

# 结构断点检测：疫情冲击、AI 爆发两个节点
# Chow 检验判断恢复曲线是否存在结构性变化

# ADF 检验：确认序列平稳性
adf_result = adfuller(df['recovery_rate'])

# 如果 LinkedIn 数据有月度粒度，做 Granger 因果
# "新质岗位增长" → "地铁客流恢复" 是否有预测力？
grangercausalitytests(panel_data[['new_quality_growth', 'metro_recovery']], maxlag=3)
```

#### 3.3 MTA 刷卡数据的时空画像

**DuckDB 聚合 8GB 文件**：

```sql
-- 站点 × 小时级聚合
SELECT
    date_trunc('hour', transit_timestamp) AS hour,
    station_complex,
    SUM(entries) AS entries,
    SUM(exits) AS exits
FROM read_csv('MTA_hourly_2022_2024.csv')
GROUP BY hour, station_complex
ORDER BY hour;
```

**Pandas 提取站点级时空特征**：

```python
# 早晚高峰比 = 早高峰(7-9点)进站量 / 晚高峰(17-19点)出站量
# 比值越接近 1 → 该站点以通勤为主
# 比值偏离 1 → 该站点混合用途（商业区、旅游区等）

morning = df[df['hour'].dt.hour.isin([7, 8, 9])].groupby('station')['entries'].mean()
evening = df[df['hour'].dt.hour.isin([17, 18, 19])].groupby('station')['exits'].mean()
stations['commute_ratio'] = morning / evening

# 2024 年各站点恢复率（vs 2022 年基线）
baseline_22 = df[df['hour'].dt.year == 2022].groupby('station')['entries'].mean()
current_24 = df[df['hour'].dt.year == 2024].groupby('station')['entries'].mean()
stations['recovery_24'] = current_24 / baseline_22
```

### 第四章：交叉验证分析

这是论文最核心的方法论章节——用三份数据的独立推断互相验证。

#### 4.1 验证逻辑一：NLP 推断 vs 刷卡行为

| 推断来源 | 方法 | 结论方向 |
|---|---|---|
| LinkedIn NLP | TF-IDF + 聚类 → A 区新质岗位占比高 | A 区应以通勤型客流为主 |
| MTA 刷卡 | 站点早晚高峰比 | A 区 commute_ratio 应接近 1 |
| **交叉验证** | 回归：`commute_ratio ~ new_quality_ratio` | 正相关 → NLP 推断被行为数据验证 |

#### 4.2 验证逻辑二：就业恢复 vs 交通恢复

| 推断来源 | 方法 | 结论方向 |
|---|---|---|
| LinkedIn | 新质行业岗位占比高 → 该行业恢复快 | 新质行业密集区地铁恢复率高 |
| MTA 日数据 | 2024 年各区域客流恢复率 | 同区域的恢复率是否确实更高 |
| **交叉验证** | 散点图 + Pearson/Spearman 相关系数 | 正相关 → 行业推断被交通数据验证 |

#### 4.3 机器学习交叉验证（xgboost + SHAP）

```python
# 特征矩阵：来自 LinkedIn NLP（行业结构）+ MTA 刷卡（站点特征）
X = stations.merge(geo_features, on='region')[
    ['new_quality_ratio', 'remote_ratio', 'job_diversity',
     'avg_salary', 'commute_ratio']
]
y = stations['recovery_24']

# XGBoost 回归：预测站点 2024 年恢复率
model = xgb.XGBRegressor(n_estimators=200, max_depth=5)
model.fit(X, y)

# SHAP 解释：哪个特征驱动了恢复？
import shap
explainer = shap.Explainer(model)
shap_values = explainer(X)
shap.plots.beeswarm(shap_values)        # 特征重要性散点
shap.plots.bar(shap_values)             # 均值条形图
```

**如果 `new_quality_ratio` 的 SHAP 值最高** → LinkedIn NLP 的行业分类确实解释了交通恢复，交叉验证成立。

### 第五章：结论与政策建议

- 新质生产力对城市交通的冲击是"结构性"而非"总量性"——总量可能已恢复，但早晚高峰分布、通勤流向已经永久改变了
- 政策建议：产业布局应与轨道交通容量协同规划
- 方法论贡献：多源时空数据交叉验证框架可推广到其他城市

## 技术栈分工总览

核心思路：**大文件入口走 DuckDB → 导出 Parquet 中间表 → 后续全部走 Pandas。**

| 步骤 | 工具 | 具体场景 |
|---|---|---|
| 8GB 大文件聚合查询 | duckdb | SQL on CSV → 导出 Parquet，内存峰值 ~200MB |
| 数据清洗与特征工程 | pandas | 中间表几万行，全流程贯通 |
| NLP 文本挖掘 | scikit-learn | TF-IDF 向量化、NMF 主题聚类、KMeans |
| 时间序列分析 | statsmodels | ADF 平稳性检验、Chow 结构断点、Granger 因果 |
| 机器学习建模 | xgboost + scikit-learn | XGBoost 恢复预测、SHAP 可解释性、RandomForest 基准 |
| 数值计算 | numpy | 数组运算、缺失值处理 |
| 可视化 | matplotlib + seaborn | 论文图表：时序图、SHAP 蜂群图、散点矩阵、热力图 |
| 进度监控 | tqdm | 大规模数据处理的进度反馈 |

5. **机器学习**：XGBoost 用 NLP 输出的行业特征预测站点恢复率，SHAP 解释证实新质岗位占比是最强预测因子
6. **交叉验证**：LinkedIn 的文本推断、MTA 刷卡的行为数据、MTA 日数据的宏观趋势——三个独立来源指向同一结论，这就是"交叉验证"
7. **工程**：8GB 文件 pandas 直接 OOM，用 DuckDB 的 `read_csv` + SQL 聚合，内存 200MB 搞定
