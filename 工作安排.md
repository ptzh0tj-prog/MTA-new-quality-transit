## 模块一：MTA 时序深化（扩充 timeseries_analysis.py）

**目标**：从 MTA 日/时数据中提取全部时间维度的证据。

### 1.1 Chow 结构断点检验

```
检验两个节点：
  - 2020-03（疫情冲击）   → 预期：显著断点（验证方法有效性）
  - 2022-12（ChatGPT 发布）→ 核心问题：AI 加速期是否有结构性变化？
```

- 对 Subway / Bus / LIRR / MNR 四条恢复曲线分别检验
- 输出：Chow 统计量 + p 值表 + 断点标注图

- 各方式恢复率的月度离散度（std / range）

- 相关性矩阵变化：疫情前 vs 疫情后，各方式间的相关系数是否变了？

### 1.3 高峰结构演化（接入 MTA 小时数据）

- 对 428 站点计算年度 `commute_ratio`（早高峰 / 晚高峰）
- 2022 → 2023 → 2024 的 commute_ratio 分布变化
- 通勤型站点（commute_ratio ≈ 1）占比是否在下降？
- 按 borough 聚合，看哪个区的通勤模式变化最大

## 模块三：交叉验证分析（新建 cross_validation.py）

**目标**：将 LinkedIn NLP 的推论与 MTA 行为证据对照，实现三层验证。

### 3.1 L1 — 结构断点验证

```
LinkedIn 推论：AI/科技岗位 2023 起爆发
       ↓
MTA 检验：  恢复曲线在 2022-12 是否存在 Chow 断点？
       ↓
结果判断：  断点显著 → LinkedIn 推论被行为数据支持
```

### 3.2 L2 — 通勤模式验证

```
LinkedIn 推论：新质行业远程率显著高于传统行业
       ↓
MTA 检验：  428 站点 commute_ratio 从 2022→2024 是否下降？
           通勤型站点占比是否减少？
       ↓
结果判断：  commute_ratio 系统性下降 → 远程办公假说被支持
```

- 按 borough 聚合 `commute_ratio` 年度变化
- KMeans 聚类站点（3-4 类：通勤型 / 混合型 / 居住型 / 商业型）
- 分析各类站点的恢复率差异

### 3.3 L3 — 运输方式替代验证

```
LinkedIn 推论：高薪新质岗位集中在曼哈顿，通勤距离拉长
       ↓
MTA 检验：  LIRR/MNR（远郊通勤铁路）恢复率是否高于 Subway/Bus？
           高收入区域的站点恢复是否更快？
       ↓
结果判断：  LIRR/MNR > Subway > Bus → 薪资-通勤距离假说被支持
```

## 模块四：机器学习验证（新建 ml_analysis.py）

**目标**：用 XGBoost + SHAP 量化各特征对站点恢复率的贡献。

### 4.1 特征矩阵构造

```
X（站点级特征）：
  - commute_ratio（通勤比）
  - entries_cv（客流波动性）
  - peak_morning / peak_evening（高峰强度）
  - avg_daily_entries（规模）
  - borough（one-hot）
  - year_over_year_change（年度变化率）

y：recovery_2024_vs_2022（站点恢复率）
```

### 4.2 XGBoost + SHAP

- 训练 XGBoost 回归
- SHAP beeswarm + bar plot
- 与 RandomForest / LinearRegression 基准对比
- 确认最强预测因子

- 选取 3-5 个代表性站点
- SARIMAX（传统时序）vs XGBoost（ML）预测精度对比
- 展示方法论选择的自觉性

## 推进顺序

| 顺序 | 模块 | 理由 |
|:--:|------|------|
| **1** | 模块一：MTA 时序深化 | 数据最扎实，已有基础，最快出成果 |
| **2** | 模块二：postings NLP | postings.csv 是新增数据，需要从头建管道 |

| **4** | 模块四：ML 验证 | 依赖模块一的站点特征 |
| **5** | 模块五：主流程 | 最后统一入口 |
